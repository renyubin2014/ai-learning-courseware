先学什么：原型能跑不等于产品能上线

AI 应用上线要面对真实流量、成本、延迟、失败、权限和监控。PM 不必会所有底层实现，但必须能判断方案边界。

部署方式

用 API、云平台、自建模型、本地脚本，成本和控制力不同。

性能体验

用户能接受的等待时间取决于任务价值和交互方式。

稳定性

模型、检索、工具、网络任何一环失败，都要有降级和提示。

核心句
AI 技术方案要同时看质量、成本、延迟和稳定性。

只看模型效果，很容易上线后翻车。

部署方式：先选合适复杂度

早期验证通常优先 API 和轻量脚本；只有在规模、隐私、成本或定制化要求明确时，才考虑更复杂的部署。

API

最快接入，适合 MVP 和中小规模应用，但依赖外部服务。

云平台

集成检索、权限、监控和部署能力，适合企业化。

自建模型

控制力强，但工程、运维和评估成本高。

本地工具

适合个人学习、离线处理、小样本验证。

成本：按完整链路算账

AI 成本不是一个模型价格。长上下文、重复调用、检索、工具、图片、语音、人工审核都会叠加。

Token

输入和输出越长，成本越高。

模型分层

简单任务用便宜模型，复杂任务用强模型。

缓存

重复问题、固定资料和中间结果可以缓存。

批处理

非实时任务可以批量处理，降低成本和峰值压力。

延迟：让用户等待得有理由

用户不是不能等，而是不知道为什么等、等多久、等完能得到什么。长任务要设计交互和架构。

短任务 同步返回，必要时流式输出。
中任务 展示进度和阶段。
长任务 异步处理并通知结果。
重复任务 缓存或预计算。
失败任务 明确错误和重试路径。

Streaming

边生成边显示，降低体感等待。

Async Job

长报告、批量分析适合异步任务。

从减少请求、减少 token、并行化、流式输出和用户体验设计理解延迟优化。

建议：把你的 MVP 链路拆成模型时间与非模型时间。

稳定性：每个外部依赖都可能失败

AI 应用依赖模型、向量库、数据库、工具 API、权限服务和网络。上线前要设计失败处理。

Rate Limit

模型或 API 有调用限制，超限要排队或降级。

Retry

短暂失败可重试，但要避免重复副作用。

Fallback

强模型不可用时切弱模型，RAG 失败时提示资料不可用。

Monitoring

监控延迟、错误率、成本、调用量和质量反馈。
