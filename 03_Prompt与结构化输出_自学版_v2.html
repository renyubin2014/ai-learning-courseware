先学什么：Prompt 是人和模型之间的产品接口

Prompt 的本质不是写得漂亮，而是把任务、上下文、约束、输出格式和验收标准交给模型。好的 prompt 能减少歧义，让输出更稳定、更可评估。

Prompt 不是魔法

同一个模型，输入任务越清楚，输出越容易稳定。模糊任务只会把不确定性转嫁给模型。

Prompt 是接口

接口要定义输入、输出、错误处理和边界。AI 产品里，prompt 就是连接用户意图与模型能力的接口层。

Prompt 要能测试

如果一个 prompt 只能靠感觉判断好坏，就还没有产品化。至少要有样本、通过标准和失败记录。

核心句
Prompt = 任务说明 + 上下文 + 约束 + 输出格式 + 验收标准。

先把这五件事写清楚，再谈模型效果。

任务说明：先把模型要做什么说清楚

一个稳定的 prompt，第一段通常不是角色扮演，而是任务定义。你要让模型知道它要解决什么问题、读哪些材料、输出给谁用。

目标

说明最终要完成的任务，例如“根据门店数据判断开店风险”，而不是“帮我分析一下”。

输入

列清楚会给模型什么材料：用户问题、表格字段、知识库片段、历史对话、工具返回。

输出用途

说明结果会被谁使用，进入什么流程。给运营看、给用户看、写入系统，要求完全不同。

边界

说明不能做什么，比如不能编造事实、不能给投资建议、没有依据时要拒答或要求补充信息。

结构化输出：让结果能被系统使用

自由文本适合阅读，不适合进入系统。只要后续要解析、排序、统计、审核，就应该优先设计结构化输出。

字段 先定义必须返回哪些字段。
类型 每个字段是文本、枚举、数字还是数组。
约束 限定长度、取值范围、是否必填。
错误 资料不足时返回什么，而不是硬填。
验收 用样本检查格式和事实是否通过。

适合结构化的场景

风险评级、线索打分、客服摘要、门店诊断、合同要点抽取、报告目录生成。

不适合强结构化的场景

早期发散、创意探索、用户访谈开放总结。此时可以先自由输出，再逐步沉淀字段。

示例：Few-shot 是给模型看标准答案

当任务标准很难一句话说清时，给模型 1-3 个高质量示例，比堆很多抽象要求更有效。

正例

展示你希望的输入和输出，让模型学习格式、语气、字段粒度和判断标准。

反例

展示不接受的输出，比如没有引用、理由空泛、字段缺失、把猜测写成事实。

边界例

展示资料不足、信息冲突、高风险问题时应如何拒答或要求补充信息。

迁移例

把你的本地生活、出行、空间数据场景写成示例，可以让 prompt 更贴近你的优势领域。

评估：Prompt 迭代不能只靠感觉

Prompt 的问题通常不是“写得不够长”，而是没有用稳定样本检查。每次改 prompt，都要知道改动解决了哪个失败模式。

评估样本

准备 20-50 条真实或近似真实任务，覆盖简单、复杂、资料不足、冲突信息和高风险场景。

通过标准

同时看格式通过率、事实正确率、引用命中率、人工修改时间和用户是否采纳。

失败模式

记录模型常见失败：编造、漏字段、答非所问、格式错、过度自信、没有拒答。

迭代方式

一次只改一个变量：任务说明、上下文顺序、输出格式、示例或参数。否则无法知道哪个改动有效。

问题 1

查看答案

为什么 prompt 不是越长越好？

参考答案

因为长 prompt 会增加成本，也可能引入冲突和噪音。好的 prompt 应该清楚、必要、可测试，而不是把所有要求堆在一起。

问题 2

查看答案

什么时候需要结构化输出？

参考答案

当输出要进入系统、被解析、被统计、被审核或被后续工具使用时，就应该优先结构化。

问题 3

查看答案

Few-shot 示例应该怎么选？

参考答案

选最能代表标准的正例、反例和边界例。示例要短、准、覆盖关键判断，不要给模型错误样式。

问题 4

查看答案

如何判断一次 prompt 迭代真的变好？

参考答案

用同一批评估样本比较格式通过率、事实正确率、引用命中率、人工修改成本和采纳率。
