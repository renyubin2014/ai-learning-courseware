模型怎样从聊天走向 Agent？

一个语言模型怎样在不被神化为“自主员工”的前提下，安全地读取信息、调用工具并完成多步任务？

学习目标能区分 Workflow、Agent、Skill 与 Know-how，画出 Agent Loop，并为状态、记忆、权限、失败恢复和人工审批设计边界。
预计时间50–60 分钟
本课产物设计一个“地图异常诊断 Agent”最小系统，标明输入、知识、工具、状态、人工审批点、停止条件与失败恢复。

学习路线 · 可直接跳转模块

边界

聊天回答与影响外部世界是两件事

聊天模型主要把输入变成文本输出；一旦系统读取私有数据、发送消息或修改记录，就需要明确的权限、工具和审计设计。

对照

聊天：输入文本→生成文本。行动系统：输入→判断→受控调用工具→外部状态变化→记录与验证。

边界

能生成一段操作建议，不等于系统已被授权执行该操作。

深入一层：定义、机制与边界

从低到高可看成四层：Workflow 固定路径，Agent 动态选路，Skill 封装可重用能力，Know-how 沉淀一项任务的判断经验与边界。

工作流

工作流用预定义路径换取可控性

当步骤稳定、风险可预见时，由代码明确规定“先做什么、后做什么”通常更容易测试和审计。

工作流

由预定义代码路径编排 LLM 与工具的系统。

类比：标准作业单

收件、分类、复核、发出，每一步都有既定顺序。

对应关系：工作流把控制权主要放在开发者定义的路径中。

类比边界：工作流并不意味着没有模型判断；它限制的是流程分支和工具编排方式。

深入一层：定义、机制与边界

Workflow 的优势是路径可穷举、易测试；当任务分支难以预先写完时，才需评估引入 Agent 的额外成本与风险。

Agent

Agent Loop 用反馈把多步任务闭环

Agent 循环的最小骨架是：规划→行动→观察→修正→结束；每一轮都应根据工具结果或环境反馈判断下一步。

Agent Loop

在停止条件前反复规划、调用工具、读取观察并更新计划的控制循环。

类比：带权限的项目助理

根据任务决定先查资料还是先计算，但只能使用获准的工具和预算。

对应关系：Agent 的差别在于动态编排，而非拥有意识。

类比边界：动态选择不应绕开权限、预算、人工审批或停止条件。

深入一层：定义、机制与边界

结束条件应同时包含成功、失败、超时、超预算和需要人工决策；否则循环可能无限重试或扩大副作用。

状态记忆

状态记录当前进度，记忆保留跨步线索

状态记录已做哪些步骤、工具返回什么和当前计划；记忆则有选择地保留跨轮次或跨任务信息。

对照

状态回答“这次任务进行到哪里”；记忆回答“以后还需要保留什么”。

边界

记忆不应无限收集私人数据；保留范围、时长、更新和删除都需明确规则。

RAG 读路径

检索→筛选→注入→生成；它把外部材料放入当前上下文，不负责执行写入动作。

深入一层：定义、机制与边界

状态应能从检查点恢复，并区分已执行的外部动作与未执行计划；否则重试时可能重复发送或重复写入。

能力

Skill 封装能力，Know-how 补上判断经验

Skill 把一组可重用步骤、工具与输入输出封装起来；Know-how 记录什么情况选哪条路、什么信号要停止。

例子

“查询道路状态并生成异常摘要”可做成 Skill；雨雪天应换用哪个数据源属于 Know-how。

边界

Skill 不一定需要 Agent，Know-how 也不应只是无法测试的长提示词。

工具调用

模型或编排器先产生工具名和结构化参数，由宿主校验与授权后才交给外部程序执行。

深入一层：定义、机制与边界

可维护的 Skill 应声明适用条件、输入合同、允许工具、成功信号和失败退路；Know-how 要通过案例与评估持续校准。

连接

MCP 是连接协议，不等于 Agent、模型或工具

MCP 为 LLM 应用与外部数据源、工具的集成规定了标准化通信方式，可涉及 host、client、server、resources、prompts 和 tools。

MCP

Model Context Protocol：用于让 LLM 应用标准化连接外部上下文和能力的开放协议。

类比：插座与插头标准

它让设备以约定方式接通，但不决定设备要完成什么任务。

对应关系：MCP 规范连接与能力暴露方式。

类比边界：MCP 不等于 Agent、模型或工具本身；是否调用、调用什么以及权限控制仍由应用设计决定。

深入一层：定义、机制与边界

MCP 可降低对接成本，但 server 提供的工具元数据仍是外部输入；host 必须实施信任边界、用户同意和最小权限。

控制

权限、Human-in-the-loop 和失败恢复是独立控制面

系统应将最小权限、高风险写入前的人工审批、参数校验、日志、超时与失败恢复放在模型输出之外。

最小控制面

允许哪些工具？哪些写入必须人工确认？如何从检查点恢复？哪些副作用必须回滚或人工处理？

边界

提示词要求“请小心”不能替代权限系统、参数验证或人工审批。

深入一层：定义、机制与边界

失败恢复不等于无限重试；对非幂等写入要保存执行回执，对不可逆操作则在执行前让 Human-in-the-loop 确认真实参数与后果。

评估

衡量端到端任务，而不是只看文笔

Agentic 系统要分别评估检索、工具选择、参数正确性、执行结果、延迟、成本和安全事件，并保留可复现日志。

端到端评估

以任务是否正确、可控地完成为目标，覆盖中间决策和最终外部结果的评估。

课程结论

适合先用稳定工作流的问题，不必为了“更智能”强行改成自主 Agent；复杂度应由任务不确定性和收益证明。

深入一层：定义、机制与边界

评估样例需要记录起始状态、每次规划与工具回执、停止原因和外部结果，否则无法定位是模型、数据、工具还是权限环节失败。

关键词学习

把核心词汇变成可检查的概念

选择一个词，比较它的释义、例子和适用边界。禁用 JavaScript 时，每个词的完整说明仍会保留在词条下方。

Workflow

释义：由预定义代码路径编排模型与工具的系统。

例子：固定经过分类、复核、写入草稿。

边界：它可包含模型判断，但路径由代码规定。

自测

不用背术语，检查你能否解释机制

先在心里作答，再展开参考答案。禁用 JavaScript 时，答案会直接显示。

参考答案：工作流由预定义代码路径编排；Agent 可动态决定过程和工具使用。

这一区分关注控制流，不以界面是否像聊天机器人来判断。

参考答案：不是。

MCP 是连接协议；它不决定是否有 Agent、用哪个模型或调用哪个工具。

参考答案：它会改变外部状态并可能造成不可逆影响。

模型输出和工具调用都可能错误，确认与最小权限形成独立控制面。

参考答案：把可检索的外部材料带入模型上下文。

RAG 不是执行动作或动态规划的同义词。

课内实验

把机制变成一次可观察的判断

完成交互后，再用下面的练习清单把结果转化为本课产物。

为地图异常诊断设计受控边界

任务：诊断地图道路状态异常。选择知识源与工具，并决定人工审批发生在高风险写入之前还是之后。

任务：诊断地图道路状态异常

知识源
工具权限

先选择知识源、工具和审批点，再检查边界。

深入一层：完成练习清单
  1. 画出地图异常诊断的 Agent Loop：规划→查询→观察→修正→结束，并为每步标出状态。
  2. 将已批准规则库与实时告警标为只读知识，将路段查询和线上修改标为不同权限的工具。
  3. 在线上修改之前放置 Human-in-the-loop，让审核者看到路段、新状态、证据和预期后果。
  4. 为超时、数据冲突、工具失败和重复写入设置停止条件、检查点和失败恢复路径。

成功信号：能说清哪些步骤只读、哪些步骤改变外部状态，并在写入前设审批、在失败后有可恢复的停止点。
