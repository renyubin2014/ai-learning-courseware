先学什么：Agent 不是会聊天的机器人

Agent 的核心不是对话，而是围绕目标持续行动：拆步骤、调用工具、观察结果、调整下一步。它比普通聊天强，也比普通聊天更难保证可靠。

一次调用

适合总结、改写、分类、抽取等单步任务。

工具调用

适合需要查数据、算价格、建工单、发请求的任务。

Agent

适合目标明确但步骤不固定，需要多轮观察和调整的任务。

核心句
Agent = 目标 + 计划 + 工具 + 观察 + 继续行动。

如果不需要持续行动，就不要为了时髦强行做 Agent。

Tool Use：让模型调用真实能力

Tool Use 是 Agent 的基础。模型本身不会查数据库、发邮件、创建订单，它需要通过工具调用外部系统。

理解意图 模型判断用户想完成什么。
选择工具 从可用工具里选合适的函数或 API。
填参数 把自然语言转成结构化参数。
执行工具 系统调用真实服务。
解释结果 模型把工具结果转成用户能理解的回答。

Function Calling

把工具描述成函数名、参数和说明，让模型输出可执行的结构化调用。

工具边界

模型只能调用被授权的工具。工具的权限、参数校验和错误处理必须由系统负责。

Agent 回路：行动后要观察结果

Agent 与普通工具调用的区别在于它会多步行动。每一步执行后，系统把观察结果返回给模型，模型再决定下一步。

目标 用户希望完成的任务。
计划 模型拆出下一步。
行动 调用工具或请求信息。
观察 读取执行结果。
迭代 继续、修正或停止。

Planner

根据目标拆出候选步骤，但计划不等于执行结果。

Action

选择一个工具或操作，并给出参数。

Observation

工具执行后的真实返回，包括成功、失败、空结果或权限不足。

Stop

达到目标、风险过高、资料不足或超过预算时停止。

可靠性：Agent 最大的问题不是聪不聪明

Agent 会行动，所以风险比聊天高。产品设计要先考虑权限、审批、幂等、回滚、日志和失败条件。

权限

不同用户、不同场景能调用的工具不同。不要把全部工具暴露给模型。

审批

涉及资金、外发、删除、批量修改等动作时，应加入 Human-in-the-loop。

幂等

同一次操作重复执行不应造成重复扣款、重复发券、重复建单。

回滚

动作失败或误操作时，要能撤回、补偿或至少记录审计链路。

边界：什么时候不要做 Agent

Agent 不是越多越好。固定流程、低不确定性、高风险动作，很多时候用普通工作流加模型调用更稳。

固定流程

如果步骤稳定，例如固定字段抽取、固定审批流，用工作流编排比 Agent 更可控。

高风险动作

资金、权限、删除、外发消息等动作，不应让 Agent 自动闭环。

目标不清

用户目标含糊时，应先澄清，而不是让 Agent 自己猜。

评估困难

如果无法定义完成标准和失败条件，就不适合直接上线 Agent。

Tool Use 和 Agent 的区别是什么？

参考答案

Tool Use 是一次或少量工具调用；Agent 是围绕目标多步行动，行动后观察结果，再决定下一步。

问题 2

查看答案

为什么高风险动作不能让 Agent 自动闭环？

参考答案

因为 Agent 可能误解目标、选错工具或参数。资金、外发、删除、批量修改等动作需要审批、日志和回滚。

问题 3

查看答案

什么场景更适合固定工作流而不是 Agent？

参考答案

步骤稳定、规则明确、失败代价高的场景更适合固定工作流加模型调用，例如固定审批、结构化抽取、合规校验。
